## 1. Один пользователь — одна острая боль

По принципу регламента (TVEP): *«один пользователь — одна острая боль»*.

**Пользователь:** Аружан, 18 лет, первый курс. Хочет волонтёрить впервые. Никого в движении не знает.

**Боль:** она не может пройти путь от **«хочу помочь»** до **«пришла на первый сбор»**. Намерение есть — моста нет. Координация локальных инициатив живёт в разрозненных чатах: информация теряется, новичку не подключиться, и порыв гаснет за один вечер.

Это не догадка, а измеримая зона: в Казахстане **350 000+** волонтёров и рост организаций **224 → 810** за 2020–2025, но единой точки входа для новичка нет (данные — из материалов направления Tech Vision).

> Платформа обслуживает три роли (волонтёр / координатор / НКО), но **северная звезда — активация волонтёра-новичка**. Всё остальное существует, чтобы сработал именно этот момент: Аружан видит «идут ещё 12 человек» только потому, что координаторы ведут на erik реальные сборы.

## 2. Как erik снимает боль — по одному барьеру

Каждый под-барьер новичка (уровень CustDev) закрыт конкретной функцией продукта:

| Страх новичка | Что снимает в erik |
|---|---|
| «Где вообще найти сбор рядом?» | Лента и карта сборов по городу |
| «Это реально или развод?» | Верифицированные НКО, виден организатор |
| «А вдруг приду — и я одна?» | **Честный прогноз явки**: видно, сколько дойдёт |
| «Мой вклад заметят?» | Часы, надёжность и бейджи в профиле |
| «Страшно регистрироваться» | Гостевой режим без аккаунта |

Ключевая находка: страх «приду одна» снимается не обещанием, а **числом** — прогнозом явки. Это и есть обоснование, почему тут нужна технология, а не ещё один чат.

## 3. Что уже работает (MVP)

- Онбординг с выбором роли (волонтёр / координатор / НКО), RU/KZ.
- Лента и карта сборов, гостевой просмотр без регистрации.
- Создание сбора → код/ссылка для чата → RSVP «Приду / Пока не знаю / Не в этот раз».
- **Дашборд координатора**: прогноз явки с интервалом неопределённости, отметка явки (с офлайн-синком), напоминания сомневающимся.
- **Штаб организатора**: заявки волонтёров (скиллы + сообщение, апрув), база волонтёров.
- Профиль волонтёра: часы, события, надёжность, бейджи, история участия.
- Рейтинг, НКО-страницы, раздел помощи (charity), сообщения, уведомления, модерация.

Полнота прототипа показана в видео (раздел вверху) и на живой ссылке.

## 4. Архитектура

Монорепозиторий из трёх независимых частей:

```
erik/
├── front/     — React 19 SPA (дизайн-система, сторы, мат-модель прогноза на клиенте)
├── backend/   — Flask REST API (SQLAlchemy, JWT, device-identity, seed демо-данных)
└── ml/        — обучаемая ML-модель прогноза явки (train / evaluate / inference)
```

### Frontend (`front/`)

- **React 19** + Create React App, **JavaScript** (без TS).
- **Tailwind CSS v4** — только токены дизайн-системы, компилируется отдельным `@tailwindcss/cli` (`src/index.css → src/output.css`); стилизация — inline + CSS-переменные.
- **Zustand 5** — 4 стора: `session / ui / gathering / platform`; оптимистичные мутации с откатом.
- **react-router-dom 7**, i18n RU/KZ (единый словарь `src/i18n.js`), анимации на CSS-keyframes.

Карта роутов:

```
/              лендинг          /c/:id         сбор — координатор (ядро)
/onboarding    выбор роли       /c/:id/check   отметка явки
/g/:code       сбор — участник  /me            мои сборы
/feed          лента            /u/:id         профиль волонтёра
/map           карта            /o/:id         НКО
/e/:id         событие          /leaderboard   рейтинг
/new           создание сбора   /charity       нужна помощь
/manage        штаб организатора /messages     сообщения
/admin         модерация        /notifications уведомления
```

### Backend (`backend/`)

- **Flask 3** + **SQLAlchemy 2** + **Flask-JWT-Extended** + **Flask-Migrate**, тесты на **pytest**, **Docker**.
- **Device-identity:** участник — это `User` с `device_id` (токен выдаётся без пароля по `X-Device-Id`); НКО/админ — тот же `User` + email/пароль. Один актор, разные привилегии.
- 8 блюпринтов: `auth · session · gatherings · guest · notifications · platform · organizer · admin`.
- Бизнес-логика вынесена в `services/`: `forecast` (аналитический прогноз), `attendance_ml` (мост к ML), `identity`, `codes`, `context`, `notifications`.

**Схема данных** (модульная, «enum»-поля валидируются в коде для дружбы с миграциями SQLite):

- Ядро: `Gathering` · `Participant` (RSVP + явка) · `GatheringCoordinator`.
- **`AttendanceRecord`** — неизменяемый журнал явки, источник истины для обучения `trust`.
- **`ForecastParams`** — синглтон-конфиг модели (BASE/ALPHA/σ), чтобы переоценивать параметры на реальных данных, а не хардкодить.
- Соц-слой: `Org · CharityRequest · Donation · Follow · Conversation · Message · Report · Application`.
- Каталоги: `User · Theme · City · Notification · Reminder · Badge`.

### ML (`ml/`) — см. [`ml/README.md`](ml/README.md)

Отдельная обучаемая модель отвечает на вопрос «придёт ли конкретный волонтёр на следующий сбор» по причинно-собранным признакам (история явки, свежесть, совпадение темы с интересами). Мост [`backend/services/attendance_ml.py`](backend/services/attendance_ml.py) грузит модель один раз и **мягко деградирует** (если модель не обучена — `available:false`, сервер не падает).

## 5. Инженерное ядро: два слоя прогноза (не обёртка над чужим API)

Прогноз явки — это **собственная модель**, а не вызов LLM. Два слоя дополняют друг друга:

**Слой 1 — аналитический** (`front/src/lib/forecast.js` = `backend/services/forecast.py`, покрыт юнит-тестами):

```
p_i = clamp( base(answer) · trust_i · ctx,  p_min, p_max )
trust_i = (came_i + α·base) / (total_i + α)      # сглаживание Лапласа, α=3
E = Σ p_i        σ = √Σ p_i(1−p_i)        интервал ≈ E ± 2σ
```

Каждая отметка явки уточняет `trust` — продукт становится точнее от использования.

**Слой 2 — обучаемый ML** (`ml/`): HistGradientBoosting / RandomForest / LogReg на журнале явки, единая `features.compute_feature_row()` для train и serve (нет train/serve skew), честный сплит **по волонтёрам** (`GroupShuffleSplit`), калибровка (ROC-AUC, Brier, log-loss). Эндпоинт `GET /api/gatherings/:id/ml-forecast` возвращает ML-оценку рядом с аналитической.

Это прямой ответ на критерий «глубина интеграции AI — кастомные цепочки вместо пустых обёрток».

## 6. Локальный запуск

**Frontend:**

```bash
cd front
npm install
npm start        # tailwind --watch + react-scripts → http://localhost:3000
npm test         # юнит-тесты мат-модели прогноза
npm run build    # прод-сборка
```

**Backend:**

```bash
cd backend
python -m venv venv && venv/Scripts/activate     # Windows
pip install -r requirements.txt
cp .env.example .env                              # заполнить секреты
flask seed-demo                                   # демо-данные (сбор PARK18 + участники)
python app.py                                     # → http://localhost:5000/api
pytest tests/ -v                                  # тесты
# либо: docker compose up --build
```

Связать фронт с локальным бэком: задать `REACT_APP_API_URL=http://localhost:5000/api` в `front/.env`. Без него фронт работает на демо-данных из `src/lib/data.js`.

**ML** (обучение модели прогноза):

```bash
cd ml
python -m venv .venv && . .venv/Scripts/activate
pip install -r requirements.txt
python train.py                 # обучить (данные сгенерирует сам)
python evaluate.py              # метрики + графики в artifacts/
python inference.py --demo      # показательные предсказания
```

## 9. Честные пометки (по регламенту)

- **Данные демонстрационные.** Живой прототип и seed используют синтетические сборы, НКО, города, рейтинги. `base`-вероятности модели взяты из первых CustDev-интервью и пересчитываются на реальных сборах (`ForecastParams`).
- **ML-эндпоинт требует обученного артефакта** в окружении бэкенда; без него отвечает `available:false` с подсказкой, а не роняет сервер.
- **Тёмная тема в приложении не делалась** — осознанное решение MVP.
- **Полная поддержка казахской кириллицы** (ә ғ қ ң ө ұ ү һ і) во всех шрифтах.
