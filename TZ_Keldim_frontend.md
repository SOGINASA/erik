## 0. Статус документа

Разделы **3–8 (дизайн-система, компоненты, motion, архитектура, стор, структура проекта)** — финальные и не зависят от результатов CustDev. Их можно реализовывать немедленно.

Разделы **1–2 и 5 (продукт и наполнение экранов)** построены на **рабочей гипотезе**, которая проверяется интервью 17–18 июля. После интервью гипотеза либо подтверждается, либо содержимое экранов переписывается — каркас, дизайн-система и сторы при этом сохраняются.

Правило пересборки: меняется **что** на экране, не меняется **как** это выглядит и как устроено.

## 1. Продукт

### Один пользователь

**Координатор локальной инициативы** — человек, который собирает людей на разовое дело (субботник, сбор помощи, мероприятие). У него нет CRM, нет НКО-бюджета и нет ассистента. У него есть чат.

Не «волонтёры вообще». Не «жители». Не «НКО». Один человек.

### Одна острая боль

Он объявляет сбор в чате. Приходит поток «+», «буду», «постараюсь». В день сбора **приходит в 3–5 раз меньше людей, чем подтвердило**, и он узнаёт об этом в момент, когда уже поздно: инвентарь закуплен на 40 человек, задачи распределены на 40, пришло 11.

Он не может отличить настоящее «+» от вежливого. Единственный инструмент — обзвон накануне, который съедает вечер и всё равно не даёт точной цифры.

**Формулировка для TVEP, стр. 1:**
> Координатор локальной инициативы не может заранее узнать, сколько людей реально придёт на его сбор, потому что подтверждения в чате ничего не значат, — и поэтому планирует ресурсы вслепую, теряя деньги, инвентарь и доверие людей, которые пришли зря.

### USP

Не «платформа для волонтёров». **Прогноз явки.** Мы не собираем людей — мы говорим координатору честное число за 24 часа до сбора и показываем, на кого именно можно рассчитывать.

### Чем это не является

- Не соцсеть. Нет ленты, нет лайков, нет профилей с био.
- Не биржа заданий. Мы не ищем людей — координатор приводит своих.
- Не замена чату. Чат остаётся. Мы — слой поверх него.

## 2. Роли и сценарии

| Роль | Устройство | Что делает |
|---|---|---|
| Координатор | Телефон + десктоп | Создаёт сбор, шлёт ссылку в чат, смотрит прогноз, отмечает явку |
| Участник | Телефон | Открывает ссылку, отвечает одним тапом, приходит, отмечается |

### Сценарий A — координатор (главный, его показываем на демо)

1. Создаёт сбор: что, где, когда, сколько рук нужно.
2. Получает короткую ссылку → кидает в свой чат.
3. Видит, как приходят ответы. Каждый ответ окрашен уверенностью: система знает историю человека.
4. За 24 часа до сбора видит **прогноз: «подтвердило 38, придёт ≈ 14 ± 3»** и список тех, кто под вопросом.
5. Одним тапом шлёт напоминание только сомневающимся.
6. На месте отмечает пришедших → данные уходят в модель, следующий прогноз точнее.

### Сценарий B — участник

1. Открывает ссылку. Без регистрации, без пароля. Имя + телефон один раз, дальше узнаёт по устройству.
2. Три кнопки, не две: **Приду · Пока не знаю · Не в этот раз**.
   «Пока не знаю» — ключевая. Именно её отсутствие в чатах и ломает всю статистику: там любой вежливый человек жмёт «+».
3. Накануне — одно напоминание с возможностью поменять ответ без объяснений.
4. На месте — отмечается кодом сбора.

**Принцип:** менять ответ должно быть так же легко, как его дать. Человек, который может безнаказанно сказать «передумал», говорит правду. Человек, которому стыдно, просто молчит и не приходит.

## 3. Дизайн-система

### 3.1 Направление

Комфортный минимализм: спокойный, плотный, с точной типографикой и сдержанной физикой движения. Никакого «дашборд-стиля» с градиентами, тенями-подушками и неоновыми акцентами.

Мир продукта — двор, весна, грабли, подъездный чат, люди в куртках. Отсюда палитра: не корпоративный синий, не техно-чёрный. Дневной свет, зелень, бумажный лист объявления на двери подъезда.

**Одна смелость на весь продукт** — «Полоса явки» (§3.7). Всё остальное вокруг неё молчит.

### 3.2 Цвет

```
--ink         #14181A   текст, основной
--ink-2       #5B6560   вторичный текст
--ink-3       #949E98   подписи, плейсхолдеры
--paper       #F4F5F1   фон приложения
--surface     #FFFFFF   карточки, модалки
--line        #E3E5DE   границы, разделители

--yard        #2F6F4F   подтверждено, «придёт», основное действие
--yard-soft   #E8F1EB   фон-подложка под yard
--maybe       #C8842B   «пока не знаю», риск, внимание
--maybe-soft  #FBF0E2
--out         #8A9490   «не в этот раз» (не красный — отказ не ошибка)
--danger      #B4332B   только деструктив: удалить сбор
```

Правил три:

1. **Отказ не красится в красный.** Человек, который честно сказал «не приду», сделал нам подарок — он не ошибка системы.
2. `--maybe` — самый заметный цвет в интерфейсе. Вся ценность продукта в зоне неопределённости, и глаз должен идти туда.
3. Тёмная тема в MVP **не делается**. Полдня работы ради нуля баллов. Записать в README как осознанное решение.

### 3.3 Типографика

Обязательное требование: полная поддержка казахской кириллицы (ә ғ қ ң ө ұ ү һ і). Проверить каждый шрифт до вёрстки, а не после.

| Роль | Шрифт | Где |
|---|---|---|
| Display | **Onest** 600 | Заголовки экранов, число прогноза |
| Body | **Golos Text** 400/500 | Всё остальное |
| Data | **JetBrains Mono** 500 | Коды сборов, время, счётчики |

Шкала (rem, база 16px):

```
display-xl  44/1.05  -0.03em   только число прогноза
display-l   30/1.15  -0.02em   заголовок экрана
title       21/1.3   -0.01em   заголовок карточки, модалки
body        16/1.5    0        основной
body-s      14/1.45   0        вторичный
caption     13/1.4    0.01em   подписи, метки
mono-s      13/1.3    0.02em   коды, время
```

Правила: никаких `font-weight` выше 600. Никакого uppercase кроме `caption` в метках статуса. Строка текста не длиннее 68 символов.

### 3.4 Сетка, отступы, радиусы

Шаг сетки — **4px**. Разрешённые отступы: 4, 8, 12, 16, 20, 24, 32, 40, 56, 72. Ничего между.

```
radius-s   8px    инпуты, чипы, бейджи
radius-m   14px   кнопки, карточки
radius-l   22px   модалки, боттом-шиты
radius-full        аватары, круглые иконки
```

Контейнер: мобильный `padding-x: 20px`. Десктоп — `max-width: 1120px`, центр, `padding-x: 40px`.

Тач-таргет — минимум **44×44px**, без исключений.

### 3.5 Тени

Тени только у того, что физически лежит выше: модалка, шит, дропдаун, липкая панель. Карточка в потоке тени **не имеет** — её отделяет фон и `--line`.

```
shadow-pop    0 1px 2px rgba(20,24,26,.04), 0 8px 24px rgba(20,24,26,.08)
shadow-sheet  0 -1px 0 var(--line), 0 -16px 48px rgba(20,24,26,.12)
```

### 3.6 Motion

Движение здесь — не украшение, а объяснение: откуда пришёл элемент и куда он уйдёт.

```
--t-fast     120ms   цвет, opacity, hover
--t-base     180ms   появление, смена состояния
--t-move     240ms   перемещение, разворачивание
--t-sheet    320ms   модалка, боттом-шит

--ease-out   cubic-bezier(0.32, 0.72, 0, 1)     всё, что приходит
--ease-in    cubic-bezier(0.4, 0, 1, 1)         всё, что уходит
--ease-soft  cubic-bezier(0.4, 0, 0.2, 1)       цвет, размер
```

Правила:

- **Ничего не выезжает больше чем на 12px.** Элемент появляется рядом с тем местом, где будет жить.
- Уход всегда быстрее прихода: `t-base` внутрь, `t-fast` наружу.
- Одновременно двигается **одна** вещь. Если двигаются две — это ошибка композиции, не повод добавить stagger.
- Stagger в списках: **20ms**, максимум 6 элементов, дальше без задержки. Иначе длинный список выглядит как заставка.
- Нажатие кнопки: `scale(0.97)`, `t-fast`. Никакого ripple.
- Число прогноза меняется **счётом**, не подменой: анимация значения 400ms `ease-out`. Это единственная «дорогая» анимация в продукте — она на главном числе, и это осознанно.
- `prefers-reduced-motion: reduce` — все transform выключаются, остаётся opacity 120ms. Обязательно, проверяется руками.

Чего не делать: parallax, blur-in, тени, реагирующие на курсор, летающие частицы, скелетоны с бегущим бликом (см. §4.9).

### 3.7 Signature: Полоса явки

Единственный запоминающийся элемент продукта. Показывается на экране сбора у координатора.

Горизонтальная полоса — не progress bar, а **состав людей**:

```
├──────────────────┼─────────────┼──────────────┤
   придут (yard)     под вопросом    отказались
        14              24              7
                        (maybe)         (out)
   └────────────┘
   прогноз ≈ 14 ± 3   ← пунктирная скоба под зоной yard
```

Поведение:

- Сегменты меняют ширину `t-move / ease-soft` при новом ответе. Не перерисовываются — **перетекают**.
- Зона `maybe` — единственная с текстурой: диагональная штриховка 2px через 6px, opacity .5. Неопределённость видно физически.
- Скоба прогноза под сегментом `yard` — пунктир, подпись `mono-s`.
- Тап по сегменту → фильтрует список людей ниже. Полоса — это ещё и контрол, не картинка.
- Первое появление: сегменты вырастают слева направо `t-move`, число прогноза считается от 0.

Если из ТЗ надо будет что-то вырезать — это вырезается последним.

## 4. Компоненты

Общее правило: у каждого компонента спроектированы **все пять состояний** — обычное, ховер, нажатие, фокус, disabled. Плюс для данных: загрузка, пусто, ошибка. Экран без пустого состояния считается несделанным.

### 4.1 Button

Варианты: `primary` (yard, белый текст) · `secondary` (surface, border line) · `ghost` (без фона) · `danger`.
Размеры: `lg` 52px (главное действие экрана) · `md` 44px · `sm` 36px.

- Hover: яркость −4%, `t-fast`.
- Active: `scale(0.97)`.
- Focus: `outline: 2px solid --ink; outline-offset: 2px`. Не убирать никогда.
- Loading: текст остаётся на месте, слева появляется спиннер 16px, ширина кнопки **не меняется**. Прыгающая кнопка — брак.
- Disabled: opacity .4, `cursor: not-allowed`, без ховера.

### 4.2 AnswerButton

Три больших кнопки ответа для участника. Высота 56px, вертикальный стек, gap 12.

Не радиокнопки. Каждая — самостоятельная поверхность: невыбранная `surface + line`, выбранная — заливка своим цветом (`yard-soft` / `maybe-soft` / нейтральная) + бордер в полный цвет + галочка справа `t-base`.

Смена ответа не требует подтверждения и не показывает предупреждение. Тап — и всё.

### 4.3 Field

Лейбл `caption` над полем, всегда видимый (плавающие лейблы — лишняя анимация, лишний баг, лишний шум).

Высота 48, `radius-s`, border `--line` → `--ink` при фокусе, `t-fast`.
Ошибка: бордер `--danger` + текст `body-s` под полем, появление opacity + 4px вверх, `t-base`. Поле **не дёргается** — shake запрещён.

### 4.4 PersonRow

Строка человека в списке: аватар-инициалы 36px (фон — детерминированный из имени, 6 приглушённых оттенков), имя `body`, под ним `caption` — история: «был 4 из 5 раз».

Справа — статус-чип. Тап → PersonSheet.
Появление новых строк: opacity + 8px вверх, stagger 20ms.

### 4.5 Chip

Статус: `придёт` · `под вопросом` · `не придёт` · `был` · `не дошёл`.
Высота 24, `radius-full`, `caption`, фон — soft-цвет, текст — полный.

### 4.6 Sheet / Modal

Мобильный — боттом-шит: снизу, `t-sheet / ease-out`, drag-to-dismiss, ручка 36×4 сверху. Подложка `rgba(20,24,26,.4)`, fade `t-base`.
Десктоп — центральная модалка: fade + `scale(0.98 → 1)`, `t-base`.

Общее для обоих: `Esc` закрывает, фокус запирается внутри, фон не скроллится, при закрытии фокус возвращается на элемент-инициатор.

### 4.7 Toast

Снизу по центру (десктоп — снизу слева), 3.5с, свайп для закрытия. Один за раз, новый вытесняет старый.
Текст в прошедшем времени и тем же глаголом, что кнопка: кнопка «Отправить напоминание» → тост «Напоминание отправлено».

### 4.8 EmptyState

Иконка 40px `ink-3`, заголовок `title`, строка `body-s` `ink-2`, кнопка действия. Пустой экран — приглашение, а не извинение.
Тексты: «Пока никто не ответил. Скиньте ссылку в чат — ответы появятся здесь.»

### 4.9 Skeleton

Однотонная плашка `--line`, пульсация opacity .6 → 1 за 1.2с. Без бегущего блика.
Скелетон показывается **только** если загрузка дольше 200ms, иначе моргает.

## 5. Карта экранов

```
/                        Вход
/new                     Создание сбора
/g/:code                 Сбор — участник
/c/:id                   Сбор — координатор (дом продукта)
/c/:id/check             Отметка явки
/me                      Мои сборы
*                        404
```

Роутинг: `react-router`. Переход между экранами — fade 120ms, без слайдов. Слайд-переходы в вебе всегда выглядят дёшево.

### 5.1 `/` — Вход

Одна колонка, вертикальный центр, максимум 400px.

Содержит: логотип-слово; строка позиционирования — **«Узнайте, сколько человек придёт, до того как это станет проблемой»**; кнопка `primary lg` «Создать сбор»; ниже `ghost` «У меня есть код» → модалка ввода кода.

Никакого маркетинга, скриншотов, «как это работает в 3 шага». Продукт объясняется первым использованием.

Загрузка страницы: логотип opacity 0→1 `t-base`, остальное stagger 40ms. Один раз, не при каждом возврате.

### 5.2 `/new` — Создание сбора

Одна страница, не визард. Визард на 4 поля — оскорбление.

Поля:

| Поле | Тип | Заметки |
|---|---|---|
| Что делаем | text | Плейсхолдер: «Субботник во дворе» |
| Где | text | |
| Когда | date + time | Нативный пикер на мобиле |
| Сколько рук нужно | number | Степпер −/+, шаг 1, дефолт 10 |
| Ваше имя | text | Только при первом сборе |

Внизу — липкая панель (`surface`, `shadow-pop`, `backdrop-blur(12px)`), кнопка `primary lg` «Создать сбор».

После создания → **модалка ShareSheet**, не редирект. Человек только что сделал вещь, надо дать её отдать.

### 5.3 `/g/:code` — Сбор, участник

Мобильный экран, ничего лишнего.

Сверху: `display-l` название, ниже строка `body-s ink-2` — дата, время, место.
Далее строка `caption`: «Нужно 10 человек · сейчас придут 4».
**Никакого прогноза участнику не показываем.** «Придёт всего 14» превращается в самосбывающееся пророчество — человек решает, что и без него справятся. Прогноз — инструмент координатора, не публичная цифра.

Центр: три `AnswerButton`.

После ответа: кнопки схлопываются в одну строку-статус «Вы ответили: Приду · изменить», `t-move`. Ниже появляется карточка с адресом и кнопкой «Открыть в картах». Никаких конфетти.

Если ответ «Пока не знаю» — под статусом строка: «Напомним за день до сбора. Ответ можно поменять в любой момент.» Это снимает тревогу и повышает честность ответов.

Состояния: сбор не найден → EmptyState + поле ввода кода. Сбор прошёл → карточка «Сбор состоялся» + сколько людей пришло.

### 5.4 `/c/:id` — Сбор, координатор

Дом продукта. **Именно этот экран показывается на демо в финале.** Мобильный — одна колонка, десктоп — две (левая 1fr контент, правая 320px — действия).

Порядок сверху вниз:

1. **Шапка.** Название `display-l`, дата/место `body-s`, кнопка `ghost` «⋯» → SettingsSheet.
2. **Блок прогноза.** Самый крупный элемент экрана:
   - `caption ink-3` — «Придёт по прогнозу»
   - `display-xl` — **14** (счёт от 0 при первой загрузке, 400ms)
   - рядом `mono-s ink-2` — «± 3»
   - строка ниже — «Нужно 10 · прогноз выше нормы» (`yard`) либо «Нужно 20 · не хватит 6 человек» (`maybe`)
3. **Полоса явки** (§3.7).
4. **Плашка действия** — появляется, только если `maybe > 0`: «24 человека не определились. Напомнить им?» + кнопка `secondary md`. Появление: opacity + 8px, `t-base`. Одна плашка, не список советов.
5. **Список людей.** Фильтруется тапом по полосе. Заголовок группы `caption ink-3` — «Придут · 14». Строки `PersonRow`.
6. **Ссылка на сбор.** Карточка с кодом `mono`, кнопки «Скопировать» и «Поделиться».

Правая колонка (десктоп): «Отметить явку» `primary lg`, «Напомнить сомневающимся» `secondary`, «Скопировать ссылку» `secondary`. На мобиле эти же кнопки — липкой панелью снизу, главная одна: «Отметить явку».

Обновление ответов — polling раз в 10с. WebSocket в MVP не нужен и не окупится на защите. Новый ответ: строка въезжает + сегменты полосы перетекают + число пересчитывается. Всё три — одновременно, 240ms.

### 5.5 `/c/:id/check` — Отметка явки

Экран для работы **на улице, в перчатке, на морозе или на солнце**. Отдельная логика вёрстки.

- Крупный поиск сверху, фокус сразу, `type="search"`.
- Список: строки **высотой 64px**, gap 8. Тап по всей строке — отметка. Не по чекбоксу.
- Отмеченная строка: фон `yard-soft`, галочка 24px справа, `t-fast`. Строка **не уезжает из списка** — иначе случайный тап не отменить.
- Сверху липкий счётчик: `display-l` «11 / 38» + `caption` «отмечено».
- Внизу `ghost` «Пришёл кто-то не из списка» → модалка добавления имени.
- Кнопка «Завершить сбор» `primary lg` → ConfirmSheet.

Экран работает офлайн: отметки в `localStorage`, синк при появлении сети. Индикатор — `caption` «Сохраним, когда появится сеть». Во дворе Петропавловска связь есть не всегда, и это тот случай, когда офлайн — не понты, а требование сценария.

### 5.6 `/me` — Мои сборы

Список карточек: название, дата, `mono` «пришло 11 из 38». Активные сверху. Кнопка «Создать сбор» — липкая внизу.
Пустое состояние: EmptyState «У вас пока нет сборов».

### 5.7 `*` — 404

`display-l` «Такой страницы нет», `body-s` «Возможно, ссылка устарела», кнопка «На главную». Без шуток и без иллюстраций.

- `ConfirmSheet` показывается только там, где отмена невозможна. Для остального — тост с «Отменить» на 5 секунд.

## 7. Zustand

Три стора, без единого god-store. Каждый — отдельный файл, экспорт селекторами.

```ts
// stores/useSessionStore.ts — кто я на этом устройстве
interface SessionStore {
  deviceId: string;              // uuid, localStorage, ставится при первом входе
  name: string | null;
  phone: string | null;
  setIdentity: (name: string, phone: string) => void;
  clear: () => void;
}
// persist: полностью
```

```ts
// stores/useGatheringStore.ts — данные сбора
type Answer = 'yes' | 'maybe' | 'no';
type Presence = 'came' | 'missed' | null;

interface Participant {
  id: string;
  name: string;
  phone: string | null;
  answer: Answer;
  presence: Presence;
  history: { total: number; came: number };   // основа модели
}

interface Gathering {
  id: string; code: string;
  title: string; place: string; startsAt: string;
  needed: number;
  status: 'open' | 'done';
  participants: Participant[];
}

interface GatheringStore {
  current: Gathering | null;
  loading: boolean;
  error: string | null;

  load: (idOrCode: string) => Promise<void>;
  poll: () => void;                                   // 10s, только при status==='open'
  answer: (participantId: string, a: Answer) => Promise<void>;   // optimistic
  mark: (participantId: string, p: Presence) => void;            // локально + очередь
  addGuest: (name: string) => void;
  finish: () => Promise<void>;
}
// persist: только current.participants[].presence + очередь синка
```

```ts
// stores/useUiStore.ts — интерфейс
type SheetKind = 'share'|'code'|'person'|'remind'|'settings'|'guest'|'confirm'|null;

interface UiStore {
  sheet: { kind: SheetKind; payload?: unknown };
  toast: { text: string; undo?: () => void } | null;
  filter: Answer | null;                  // выбранный сегмент полосы
  openSheet: (kind: SheetKind, payload?: unknown) => void;
  closeSheet: () => void;
  showToast: (text: string, undo?: () => void) => void;
  setFilter: (f: Answer | null) => void;
}
// persist: нет
```

Правила:

- **Прогноз в сторе не хранится.** Это производная — селектор `selectForecast(state)`, мемо. Хранить вычисляемое = гарантированный рассинхрон на демо.
- Компонент подписывается **на поле, не на стор**: `useGatheringStore(s => s.current?.title)`. Подписка на весь стор = перерисовка всего экрана на каждый polling.
- Все мутации — optimistic, с откатом и тостом «Не удалось сохранить · Повторить».
- Никакой бизнес-логики в компонентах. Компонент читает и вызывает — считает стор и селекторы.

## 9. Мат-модель прогноза (TVEP, стр. 3)

Это ядро проекта. Не ИИ-обёртка — собственный алгоритм, который можно защитить на Q&A.

Для участника *i* вероятность прихода:

```
p_i = clamp( base(answer_i) · trust_i · ctx , 0.02, 0.98 )

base:   yes = 0.62   maybe = 0.24   no = 0.02        (стартовые, из CustDev)

trust_i = (came_i + α·base(answer_i)) / (total_i + α),   α = 3
          — сглаживание Лапласа: новичок без истории не ломает прогноз

ctx  = w_weather · w_lead · w_weekday                   (множители контекста)
```

Прогноз и разброс:

```
E  = Σ p_i
σ  = sqrt( Σ p_i·(1 − p_i) )       — сумма Бернулли, интервал ≈ E ± 2σ
```

Что здесь есть для жюри:

- **Обучение на своих данных**: каждая отметка явки обновляет `trust_i`. Продукт становится точнее от использования — это и есть ответ на «а чем вы лучше опроса в чате».
- **Честная неопределённость**: показываем ± , а не одно число. Координатор видит риск, а не обещание.
- **Калибровка**: `base` берутся из первых интервью и пересчитываются на реальных сборах. Если данных нет — синтетика, **честно указанная в README и на защите** (это прямо разрешено регламентом).
- **Тесты** на `model.ts` — единственные тесты в проекте. Тесты на кнопках никого не впечатлят, тесты на модели показывают, что вы понимаете, где у продукта сердце.
